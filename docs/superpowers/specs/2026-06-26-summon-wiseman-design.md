# `/summon-wiseman` 설계 문서 (v2 — 순수 LLM 위키)

- 작성일: 2026-06-26
- 상태: 설계 확정 대기 (사용자 리뷰)
- 제품명: **Wiseman** — "코딩하기 전에 여쭤보는 현자(wise man)"
- 한 줄 요약: 현재 프로젝트를 분석·심층 조사하여, **그 코드베이스 전용 LLM 위키**(카르파시식, 순수 마크다운)를 만들고 유지하는 Claude Code 플러그인. 코딩 전에 에이전트가 위키에 "여쭤보고", 새로 배운 건 위키에 되먹여 **쓸수록 똑똑해진다.**

> **v1 → v2 변경 요지**: MCP 서버 / SQLite / FastMCP / uv 런타임을 **전부 제거**. 카르파시 "LLM 위키" 모델(https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)을 코드베이스에 적용한 **순수 마크다운 + 스킬** 구조로 전환. 검색은 에이전트의 `Read`/`Grep`/`Glob`로 수행.
>
> 네이밍 노트: `wiseman`은 확정 제품명. 저장소 디렉토리명 `teachers-mcp`는 가명 — 추후 리네임 가능(코스메틱).

## 1. 목적과 핵심 가치 기준 (불변)

Wiseman은 **"우리 코드베이스에 딱 맞는 LLM 위키"** 를 만든다. 코딩 중 Claude가 행동 전에 위키에 "여쭤보면", 그 프로젝트의 실제 스택·버전·관례·함정에 맞는 답을 얻는다.

### 절대적 성공 기준 (Non-negotiable)

> **위키에 물어서 얻은 답은, 모델이 자기 내부 지식만으로 짜는 것보다 반드시 더 나아야 한다.**

따라서 위키의 모든 페이지는 다음을 만족해야 한다:

1. **최신성** — 웹 심층 조사로 현재 버전 기준 정보를 수집
2. **버전 핀(pin)** — 프로젝트에 실제로 설치된 버전에 고정
3. **출처(source) 보유** — 공식 문서 URL / 설치 코드 경로 / `derived-principle` 명시
4. **프로젝트 고유성** — 일반론이 아니라 이 프로젝트의 스택/아키텍처/관례 반영
5. **비자명성** — 뻔한 내용이 아니라 함정·안티패턴·관례 중심

이 기준은 빌드 시 **품질 게이트**(§9)로, 운영 중 **`lint` 연산**(§7-3)으로 강제한다.

## 2. 핵심 모델: 카르파시 LLM 위키를 코드베이스에 적용

카르파시의 3계층 / 3연산을 그대로 차용한다.

| 카르파시 | Wiseman 대응 |
|----------|--------------|
| ① 원본 소스(불변) | 공식 문서, 설치된 라이브러리 코드, 프로젝트 자체 코드 |
| ② 위키 (LLM 소유 마크다운) | `.wiseman/wiki/` — 라이브러리·관례·패턴·아키텍처 페이지 |
| ③ 스키마 (CLAUDE.md식 설정) | `.wiseman/schema.md` (CLAUDE.md에 `@import`로 상시 로드) |
| 연산: `ingest` | summon-wiseman / 이후 소스 추가 |
| 연산: `query` | 코딩 전 위키 열람 + **유용한 종합을 페이지로 되먹임(복리)** |
| 연산: `lint` | 모순·고아·낡은 주장 점검 |

**비전제(non-goal)**: MCP 서버, 영속 인덱스 DB, 시맨틱 검색. 모두 향후 옵션이며 MVP엔 없다. 검색은 Claude Code의 파일 도구로 충분하다.

## 3. 컨텍스트 사용 모델 (오해 방지)

위키 전체를 컨텍스트에 넣지 **않는다**. 코드베이스를 다루듯 한다.

| 무엇 | 컨텍스트에? |
|------|------------|
| `schema.md` (지도+규칙) | **항상 로드** (CLAUDE.md `@import`) — "위키 구조와 query/ingest/lint 사용법" |
| `index.md` (목차) | 필요시 훑기 |
| 개별 위키 페이지 | **관련된 것만 그때그때** (`Read`/`Grep`) |
| 위키 전체 | **절대 통째로 안 넣음** |

상시 상주는 **작은 "사용법 지도"** 뿐이고, 내용은 디스크에 있다가 관련 페이지만 끌어온다.

## 4. 아키텍처

### 레이어 1 — 플러그인 (배포물, 한 곳에서 관리)

`wiseman` 플러그인이 번들하는 것:

- **스킬들** (마크다운 지시문) — `summon-wiseman` 등 (§6, §7)
- **`schema.md` 템플릿** — 위키 운영 매뉴얼의 원본

**런타임 코드 없음.** 서버·DB·언어 런타임 의존 없음. 빌드/유지 작업은 전부 에이전트가 스킬을 따라 파일 도구로 수행한다. → `/plugin install wiseman` 은 스킬만 설치하면 끝(MCP 자동등록 불필요).

### 레이어 2 — 프로젝트별 산출물 (마크다운 위키)

```
대상 프로젝트/
├── .wiseman/
│   ├── schema.md              ← 위키 구조·워크플로 정의 (지도+규칙)
│   └── wiki/
│       ├── index.md           ← 카탈로그/목차
│       ├── log.md             ← append-only 연산 기록
│       ├── libs/              ← 라이브러리별 베스트 프랙티스 (예: fastapi-auth.md)
│       ├── conventions/       ← 프로젝트 고유 관례
│       ├── patterns/          ← 언어/클린코드 패턴·안티패턴
│       └── architecture/      ← 프로젝트 아키텍처
└── CLAUDE.md                  ← `@.wiseman/schema.md` 한 줄 추가됨
```

전부 마크다운 = **단일 진실 소스**. git 친화적(diff/리뷰/팀 공유), Obsidian 등으로 사람이 직접 열람 가능.

## 5. 위키 페이지 사양

### 5-1. 페이지 frontmatter

```markdown
---
title: FastAPI 인증 — 의존성 주입 기반
kind: library_doc        # library_doc | project_convention | clean_code | language_pattern | architecture
library: fastapi         # 해당 시
version: 0.115.2         # 핀된 설치 버전
source: https://...      # URL | 코드 경로 | derived-principle
confidence: high         # high | medium | low
updated: 2026-06-26
tags: [auth, di, security]
---

본문 (마크다운). 관련 페이지는 [[conventions/error-handling]] 처럼 상호링크.
```

### 5-2. 상호링크 & 특수 파일

- **상호링크**: `[[상대경로]]` (Obsidian 호환) 또는 일반 마크다운 링크. `Grep`으로 추적 가능.
- **`index.md`**: 전체 페이지 카탈로그(분류별 링크 목록). 새 페이지 추가 시 갱신.
- **`log.md`**: append-only. 형식 `## [YYYY-MM-DD] <op> | <title>` — 파싱·이력 추적용.

## 6. 스키마 계층 & CLAUDE.md 주입

`schema.md`는 **에이전트가 위키를 쓰는 법**을 정의한다(내용이 아니라 사용법):

- 위키 디렉토리 구조와 각 폴더의 용도
- **query 워크플로**: 코딩/스킬 사용 전, 관련 페이지를 먼저 `Grep`/`Read`로 확인하라
- **복리(fileback) 워크플로**: 조사·종합으로 가치 있는 새 지식을 얻으면 페이지로 적재 + `log.md` 기록
- **ingest 워크플로**: 새 소스 흡수 시 관련 페이지 갱신
- **lint 워크플로**: 모순/고아/낡음 점검

**주입 방식**: `/summon-wiseman`이 프로젝트 `CLAUDE.md`에 `@.wiseman/schema.md` **import 한 줄**을 추가(없으면 CLAUDE.md 생성). 이로써 매 세션 schema가 상시 로드되어, 에이전트가 "코딩 전 위키 확인 + 배운 것 되먹임"을 자동 습관화한다. import 한 줄만 넣으므로 CLAUDE.md 오염 최소.

## 7. 세 가지 연산

### 7-1. `ingest` (지식 흡수) — `summon-wiseman`이 최초 대량 수행

§8의 빌드 파이프라인 참조. 이후에도 새 라이브러리 추가/문서 갱신 시 동일 절차로 증분 흡수.

### 7-2. `query` (열람 + 복리 되먹임)

1. 에이전트가 작업 전 `schema.md` 가이드에 따라 위키에서 관련 페이지 검색·열람
2. 위키에 답이 있으면 그대로 활용
3. **없거나 부족하면** 즉석 조사 → 답을 내고 → **가치 있으면 새 페이지로 적재 + `log.md` 기록** (= 복리: 다음엔 위키가 안다)

이 복리 루프가 "현자가 쓸수록 똑똑해진다"의 실체다.

### 7-3. `lint` (건강 점검)

- **모순**: 같은 주제 페이지 간 상충 주장 탐지
- **고아**: 어디서도 링크되지 않은 페이지
- **낡음**: `version`이 현재 설치 버전과 어긋난 라이브러리 페이지, 오래된 `updated`
- 결과를 사용자에게 보고, 갱신/병합/삭제 제안

## 8. `summon-wiseman` 스킬 (최초 빌드)

### 8-1. 분기 판단

- 의존성 매니페스트(`pyproject.toml`, `package.json` 등) **있으면** → 기존 프로젝트 흐름
- 비어있으면 → 새 프로젝트 흐름

### 8-2. 기존 프로젝트 흐름 (추출 파이프라인, 순서 의미 있음)

1. **의존성 식별** — 매니페스트+락파일에서 라이브러리와 **설치된 정확한 버전** 수집
2. **웹 심층 조사** — 각 핵심 라이브러리의 공식 문서+베스트 프랙티스를 **병렬 서브에이전트**로 심층 검색 (`deep-research` 스킬 있으면 활용)
3. **공식 문서 부실 시 폴백** — 설치된 실제 라이브러리 코드(site-packages/node_modules) 열람
4. **일반 클린코드 원칙** 고려
5. **언어별 안티패턴/베스트 프랙티스** 고려
6. **프로젝트 자체** — 기존 아키텍처·코드 관례·CLAUDE.md 반영(참고 수준)
7. **위키 작성** — 위 지식을 페이지로 정제 작성 + `index.md`/`log.md` 생성 + `schema.md` 배치 + CLAUDE.md에 import 주입

### 8-3. 새 프로젝트 흐름

1. **브레인스토밍** — 무엇을 만들지, 어떤 스택을 쓸지 사용자와 결정
2. 확정 스택에 대해 8-2의 2~7 동일 수행(6은 계획된 아키텍처 기준)

### 8-4. 병렬화

라이브러리/토픽별 독립이므로 조사 단계는 **병렬 서브에이전트 팬아웃**(`dispatching-parallel-agents`). 각 에이전트가 한 라이브러리/토픽을 맡아 구조화된 페이지 초안을 반환.

## 9. 품질 게이트 (§1 강제)

위키 확정 **전** 자체 검증:

1. **출처 검사** — `library_doc` 페이지는 `source` 필수. 없으면 폐기 또는 `derived-principle` 명시.
2. **버전 핀 검사** — 라이브러리 페이지는 `version` 필수(설치 버전 일치).
3. **비자명성 스폿체크** — 표본 페이지를 "내부 일반 지식과 다른가? 더 나은가?" 검토하는 서브에이전트 패스. 일반론뿐이면 재조사.
4. **프로젝트 고유성** — `project_convention`/`architecture` 페이지 최소 1개 이상.

결과를 summon 종료 시 사용자에게 요약 보고(페이지 수, 라이브러리 커버리지, 폐기 항목).

## 12. 테스트 전략

- **스킬 흐름** — 기존/새 프로젝트 분기 판단(픽스처 기반)

- **품질 게이트** — 출처/버전 없는 페이지가 게이트에 걸리는지

(런타임 코드가 거의 없으므로 테스트는 산출물·흐름 검증 중심.)

## 13. MVP 범위

**포함:**
- `wiseman` 플러그인 (스킬 + schema 템플릿, 런타임 코드 없음)
- `summon-wiseman` 스킬 (기존/새 프로젝트 분기, 병렬 조사, 품질 게이트)
- 마크다운 위키 산출물 (`.wiseman/wiki/` + index/log + 페이지 frontmatter)
- `schema.md` + CLAUDE.md `@import` 주입
- 복리(query+fileback) & lint 워크플로 (schema.md에 정의, 에이전트가 파일 도구로 수행)

**제외 (향후):**
- 검색 가속용 MCP/인덱스 (규모 커질 때)
- 벡터 임베딩 / 시맨틱 검색
- Obsidian 외 별도 편집 대시보드
- 저장소 `teachers-mcp` → `wiseman` 리네임

## 14. 열린 질문 (구현 계획 전 확정 필요)

1. **재-summon / 갱신 정책** — 기존 위키가 있을 때: 전체 재빌드 vs 증분 갱신(권장) vs 사용자 선택?
2. **조사 깊이/비용** — 라이브러리당 서브에이전트 몇 개? 토큰 예산 상한?
3. **핵심 라이브러리 선정** — 전체 의존성 vs 직접 의존성만 vs 사용자 선별?
4. **`.wiseman/` git 커밋 기본값** — 마크다운이라 팀 공유(커밋)가 자연스러움. 기본 커밋 권장? `.gitignore` 옵션?
5. **복리 자동성** — query 중 새 페이지 적재를 **자동**으로 할지, 사용자 확인 후 할지?
6. **개별 슬래시 커맨드 여부** — ingest/lint를 `schema.md` 워크플로로만 둘지, 별도 스킬(`/wiseman-lint` 등)로도 노출할지?
